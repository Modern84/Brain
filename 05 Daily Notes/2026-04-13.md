# 2026-04-13

## ProForge5 — Großer Config-Durchbruch

### TMC UART Problem gelöst ✅

**Ursache gefunden:** Die printer.cfg hatte `[tmc2209]` mit UART-Pins konfiguriert, aber die Treiber auf dem Board sind **TMC5160** (SPI). Klipper hat versucht UART zu sprechen — die Treiber sprechen nur SPI. Kein Hardware-Defekt, kein Jumper-Problem — reines Config-Thema.

**Lösung:** printer.cfg komplett auf die offizielle MakerTech GitHub-Config umgestellt (v1.0.0, 17.03.2026):
- `[tmc5160]` mit `cs_pin` und `spi_bus: spi1` statt `[tmc2209]` mit `uart_pin`
- Korrekte dir_pins invertiert (`!PF12`, `!PG1`)
- Quad-Z (4 Motoren) auf Slots 5-8 mit korrekten Pins
- Korrekte Ströme: 2.2A X/Y (TMC5160), 1.1A Z (TMC2209)

**Ergebnis:** `DUMP_TMC` alle 6 Treiber antworten. `STEPPER_BUZZ` X und Y funktionieren.

### Offizielle GitHub-Config als Basis

Repo gefunden: [Makertech3D/ProForge-5](https://github.com/Makertech3D/ProForge-5). Enthält printer.cfg, macros.cfg, eddy.cfg, Toolhead-Configs und mehr.

Neue Config-Dateien auf dem Pi:
- **printer.cfg** — offizielle Basis + EBB36/U2C/Eddy Anpassungen
- **macros.cfg** — komplette offizielle Toolchanger-Macros
- **eddy.cfg** — angepasst für Eddy Coil via I2C am EBB36 (statt USB-MCU)
- **mainsail.cfg** — Mainsail-Standard

## Session 2 — Endstops, Homing, Netzwerk

### Y-Endstop gefixt ✅

Pin-Zuordnung war falsch. Config hatte `^!ebb:PB6`, aber der Endstop hängt physisch auf `PA15` am EBB36 (laut Pinout-Diagramm: ENDSTOP = GND + PA15). Außerdem kein Pullup/Invertierung nötig — NO-Schalter.

**Fix:** `endstop_pin: ebb:PA15` (ohne `^`, ohne `!`)

Dock-Sensor auf `ebb:PB6` umgestellt (war vorher fälschlich auf `PA15` Octopus).

### Homing X + Y ✅

- `G28 X` — funktioniert, fährt sauber zum Endstop
- `G28 Y` — funktioniert nach Pin-Fix

### CAN-Bus Auto-Start ✅

Alte `interfaces.d/can0` Config war unzuverlässig (Race-Condition mit gs_usb). Ersetzt durch systemd-Service `/etc/systemd/system/can0.service` mit 5s Delay. Überlebt Reboot.

### WLAN Auto-Reconnect ✅

iPhone-Hotspot "iP" verbindet nach Reboot nicht automatisch. Systemd-Timer eingerichtet, versucht alle 60s Reconnect: `/etc/systemd/system/wifi-reconnect.timer`

### Tailscale neu eingerichtet ✅

Pi war auf altem Account `Modern84@`, Mac auf `modern3b@` — konnten sich nicht sehen. Pi auf `modern3b@` umgestellt. Neue Tailscale-IP: `100.90.34.108`.

## Session 3 — Eddy Coil Kalibrierung (blockiert)

### Problem: I2C BUS_TIMEOUT bei Kalibrierung

Eddy Coil hängt per I2C (PB3/PB4) am EBB36 über CAN-Bus. Klipper startet sauber, Sensor wird erkannt. Aber `PROBE_EDDY_CURRENT_CALIBRATE` → ACCEPT crasht **jedes Mal** mit:
```
MCU 'ebb' I2C request to addr 42 reports error BUS_TIMEOUT
```

### Was alles getestet wurde (alles ❌)

| Versuch | Ergebnis |
|---|---|
| Hardware I2C `i2c3_PB3_PB4` | BUS_TIMEOUT |
| Software I2C PB3/PB4 | Invalid ldc1612 id (0,0) |
| Hardware I2C `i2c3_PA7_PA6` | START_NACK (falscher Pin) |
| Software I2C PA7/PA6 | START_NACK |
| `i2c_speed: 100000` | BUS_TIMEOUT |
| `data_rate: 50` (Code-Patch) | BUS_TIMEOUT |
| `txqueuelen 256` | BUS_TIMEOUT |
| Delays zwischen I2C-Writes (Code-Patch) | BUS_TIMEOUT |
| `reg_drive_current: 16` manuell | BUS_TIMEOUT |
| Dummy-Kalibrierungsdaten manuell | NoneType error (falsche Frequenzen) |
| KlipperScreen deaktiviert | BUS_TIMEOUT |

### Ursache

CAN-Bus kann die kontinuierliche I2C-Sensor-Datenstreaming während der Kalibrierung nicht handeln. Bekanntes Problem (BTT Eddy GitHub Issue #153). BTT dokumentiert den Eddy Coil via EBB36 CAN quasi nicht.

### Lösung (Hardware nötig)

**Option A:** 2x 4.7kΩ Pull-Up Widerstände auf PB3→3.3V und PB4→3.3V am EBB36 löten. Eddy Coil und EBB36 haben zu schwache/keine Pull-Ups.

**Option B:** USB Eddy reparieren (abgerissener Stecker, Datenleitungen nachlöten). Wird per USB erkannt (Strom) aber keine Daten (`lsusb` zeigt kein RP2040).

**Option C:** USB Eddy neu kaufen (~18€).

### Weitere Erkenntnisse

- EBB36 ist v1.2 (nicht Gen2) — I2C auf PB3/PB4 bestätigt
- MakerTech nutzt USB Eddy, nicht Coil — Original-Setup hat dieses Problem nicht
- Eddy Coil Setup wurde von Gemini empfohlen, ohne die CAN-Limitierung zu berücksichtigen
- BTT GitHub Issue #71 zeigt funktionierende Config mit gleichen Pins — Pull-Ups machen den Unterschied
